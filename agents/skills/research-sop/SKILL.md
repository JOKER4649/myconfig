# 研究 SOP

## 為什麼委派

本專案的搜尋權限已鎖定:只有 `@explore` 與 `@oracle` 兩個 subagent 能使用 websearch、context7、grep_app 等搜尋與推理工具。主 agent 需要外部資訊或高階判斷時,必須委派,無法直接執行。

這是刻意設計,目的:

- **隔離大量搜尋結果與網頁內容**,保護主 agent 上下文品質
- **降低 token 成本**(subagent 通常使用較便宜的模型)
- **防止外部內容的 prompt injection** 攻擊
- **可平行委派**多個 subagent 加速

## 任務分流:事實 vs 判斷

主 agent 收到任務時,**先判斷性質再決定委派對象**。`@explore` 與 `@oracle` 是不同任務的並行選項,**不是先後階層**。

| 性質 | 特徵                                     | 委派對象                                       |
| ---- | ---------------------------------------- | ---------------------------------------------- |
| 事實 | what / where / how it works / 客觀描述   | 主 agent 或 `@explore`(見「自行處理 vs 委派」) |
| 判斷 | should I / which is better / 推薦 / 比較 | `@oracle`                                      |

常見誤判:

- 把「比較兩個方案」直接丟給 `@explore` → 會被拒絕(explore 不下結論)
- 把「某個 API 的簽名」丟給 `@oracle` → 浪費 oracle 推理在事實上
- 把判斷問題包裝成事實問題(用「哪個更適合 X」偷渡)→ 應明確標示是判斷

如果任務混雜(例如「查 A 的事實 + 比較 A 跟 B」),先拆成兩塊:**事實派 `@explore`,判斷派 `@oracle`**(見「協作模式」)。

## @explore 路徑

### 自行處理 vs 委派

事實查詢不是單純的「派 / 不派」二元,而是 **目標明確度 × 內容量** 兩個維度的判斷。

|                | 量可控        | 量可能失控      |
| -------------- | ------------- | --------------- |
| **目標明確**   | 主 agent 直接 | 委派 `@explore` |
| **目標不明確** | —             | 委派 `@explore` |

具體場景:

- **本地程式碼搜尋**(glob / grep / read):目標明確 + 量可控 → 主 agent 直接。工具已相當高效。
- **已知 URL / 已知 context7 lib,頁面小**:目標明確 + 量可控 → 主 agent 直接(webfetch / context7)。
- **已知 URL 但頁面可能很大**(整個 docs site、整個 framework 的 API 清單):目標明確但量可能失控 → **委派 `@explore`**,讓 subagent 過濾後只回傳重點,避免污染主 agent context。
- **未知來源 / 開放式搜尋**:目標不明確 → 委派 `@explore`。

底層取捨:

- 直接做的成本:context 被撐爆、稀釋注意力、prompt injection 風險
- 委派的成本:多一道 subagent 的 token 與延遲
- 當「量失控的風險」>「委派的成本」時,應委派;否則直接做更划算
- 不確定時寧可委派 — 失控的代價通常高於多一道 subagent

### 何時拆成多個 subagent

當任務包含多個獨立研究對象時,主 agent 應主動拆成多個 `@explore` 並行,而不是把所有對象塞進一個 prompt。

**應拆**(符合任一即拆):

- **獨立對象**:主要研究對象之間沒有概念重疊(Trigger.dev vs Mastra — 兩個完全不同的產品)
- **獨立來源**:每個對象的主要事實來源不重疊(不同 repo、不同官方 docs)
- **可獨立完成**:每個對象各自構成「對象 / 動作 / 目的 / 判準」四要素,單獨委派也合理
- **無中間依賴**:各方向探索結果互不引用、不需等其他方向先完成

**不應拆**(符合任一即不拆):

- **比較 / 對比任務**:任務本質是 A vs B,需同一視角交叉檢視 — 整個任務派 `@oracle`
- **統一指標**:所有對象必須用同一組指標評估
- **單一輸出**:所有資料組合成一份結構化輸出,缺一不可

**拆解範本:複合研究任務**

典型結構:

```markdown
## 任務背景

(共用 context)

## 調查目標 A:<對象 1>

(子問題)

## 調查目標 B:<對象 2>

(子問題)

## 輸出要求

(回傳內容)
```

主 agent 處理流程:

1. 識別 N 個獨立對象(例:N=2)
2. 抽出任務中「比較 / 判斷 / 推薦」元素 → 留給 `@oracle` 或主 agent
3. 對每個對象組合:`任務背景 + 該方向子問題 + 該方向的輸出期望` → 一個 `@explore` prompt
4. 同時 spawn N 個 `@explore`
5. 收到結果後,主 agent 整合,送 `@oracle` 做比較(若需要)

**拆解反模式**:

- 多個獨立對象塞同一 prompt,期待 explore 自己拆(會被同一 context 限制、視角混淆)
- 比較任務強行拆分(失去交叉參照)
- 為拆而拆,把可獨立完成的小問題硬拆多次(增加來回成本)

### 寫出好的 subagent prompt

四要素框架(沿用 `@explore` 的 FAIL 檢查):

| 要素 | 說明         | 缺漏會怎樣                      |
| ---- | ------------ | ------------------------------- |
| 對象 | 具體主題     | explore 無法聚焦,搜尋方向散亂   |
| 動作 | 要做什麼     | 結果可能「對但不是你要的」      |
| 目的 | 為什麼要知道 | explore 不知道深度 / 廣度該到哪 |
| 判準 | 怎樣算完成   | 無法判斷什麼時候該停止          |

缺任一要素即為模糊,`@explore` 會 FAIL,需要重派。

**措辭風格**(`@explore` 內部搜尋引擎特性):

- **語意搜尋(Exa 型)**:具體術語、API 名稱、錯誤訊息原文 — 適合「精準命中」
- **傳統搜尋(Brave 型)**:開放描述、多種可能 — 適合「探索覆蓋」

明確答案用前者;探索比較用後者。`@explore` 內部已實作引擎選擇,主 agent 不需指定,理解這個特性有助於寫出精準的 prompt。

`@explore` 的工具選擇 / 視角清單 / 證據標準等細節見 `explore.md`。

### 平行委派

確認應拆後(見上節),主 agent 在同一訊息發出 N 個 task 呼叫,各自獨立運行,結果回傳後主 agent 整合。

若任務包含「事實 + 判斷」混雜,事實派 `@explore`,判斷留給 `@oracle`(見「協作模式」)。

### 反模式

- 給 `@explore` 過於模糊的 prompt → FAIL 重派
- 預期 `@explore` 給建議或方案 → 它只回傳可驗證事實
- 讓單一 `@explore` 負責太多不相關的查詢方向(應拆不拆)
- 判斷問題包裝成事實問題(用「哪個更適合 X」偷渡判斷)
- 把本可定位的查詢(已知 URL / 已知檔案)也派 `@explore`,浪費一道 subagent
- 大體積查詢直接做,污染主 agent context(`webfetch` 整個 docs site、`context7` 拉整個 framework 的所有 API)— 應該委派 `@explore` 過濾後只回重點

### 結果彙整

`@explore` 回傳的是結構化事實,含來源連結與屬性標籤(時效 / 驗證 / 事實性 / 來源類型)。

收到後:

1. 閱讀各 subagent 回傳的事實
2. 交叉比對來源可信度(官方 docs > 知名社群 > 個人部落格)
3. 確認事實覆蓋符合原本「判準」,缺漏時決定是否重派
4. 整合成符合使用者需求的回答
5. 引用來源連結,保持可驗證性

若發現任務本質需要判斷(例如使用者後續追問「那該選哪個」),把事實 + 問題送 `@oracle`。

## 兩個 subagent 的協作

`@explore` 與 `@oracle` 是不同任務的並行選項。常見協作模式有三種:

### 純事實

直接派 `@explore`,事實即答案。

例:「React 19 use() hook 的 API 簽名是什麼」

### 純判斷

直接派 `@oracle`,需附上完整上下文(任務背景、使用者需求、已知選項等)。

例:「我要做 X、Y、Z 三種情境的個人專案,該用 SQLite 還是 Postgres」

### 事實 + 判斷

典型 pipeline:

1. 主 agent 拆解任務,識別事實部分 + 判斷部分
2. 派 N 個 `@explore` 並行蒐集事實
3. 主 agent 整合事實
4. 把事實 + 判斷問題送 `@oracle`
5. `@oracle` 回傳建議,主 agent 包裝給使用者

這是處理「深度比較 + 推薦」類任務的標準流程。

注意:不要為了「讓 oracle 看起來很忙」硬送 oracle。純事實就別送;純判斷且上下文已備齊,可直接送,不必先過 explore。
