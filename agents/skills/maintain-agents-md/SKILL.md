## 適用範圍

此 SKILL 處理的是**專案根目錄**的 `AGENTS.md`（給該 repo 內的 AI 編碼 agent 看的運作守則），不是 `~/.config/opencode/AGENTS.md`（用戶級全域規則）。兩者職責分離：全域規則管所有專案通用行為，專案級 `AGENTS.md` 只描述此專案的特殊性。

## 內容原則

- 給 agent 看的**運作守則**，不是給人類的 README；不寫教學、不寫廣告、不寫時效資訊
- 每行必須有資訊密度；agent 不靠 context 就能推論的內容（commit 前跑 lint、常見的 git 流程）不必寫
- 不與 `README.md` / `CONTRIBUTING.md` 重疊；README 講「這專案是什麼」，AGENTS.md 講「在這專案工作要注意什麼」
- 只寫變動頻率低的內容；太常變的東西（例如當前 sprint 待辦、版本號）會快速過時，會成為噪音
- 不要寫敏感資訊（token、內網位址、密鑰路徑個資等）

## 建議章節

依照必要性取捨，不是每個專案都需要全部：

- **專案概述**：一到三句話講清楚這專案的目的與邊界
- **專案結構**：目錄樹，每個子目錄附 5-15 字描述（用 `tree` 或 `ls` 產出後精簡）
- **慣例**：命名、檔案組織、commit message 風格、模組邊界等
- **注意事項**：容易踩坑的地方、危險操作、非顯而易見的限制、`.gitignore` 排除但不查 git log 看不到的東西
- **工具特定子節**：例如 `commands/`、`scripts/` 的特殊使用方式（若僅在該子目錄內有效）

不要的章節：安裝步驟、快速開始、貢獻指南、授權聲明——這些屬於 README。

## 工作流程

### 1. 決定範圍

- **建立**：專案無 `AGENTS.md` 或用戶明確要求從頭寫
- **局部更新**：新增/移除單一目錄、慣例微調、注意事項新增/移除
- **全檔重寫**：使用者明確要求、或專案結構大幅變動
- **預設採用局部更新**，除非變動已影響超過約 40% 的內容或使用者明確要求重寫

### 2. 盤點現況

- 用 `ls`、`tree`、glob 讀取當前目錄結構
- 對照既有 `AGENTS.md` 的「專案結構」段落
- 檢查 `.gitignore`、`README.md`、根目錄的建構/設定檔（`package.json`、`pyproject.toml`、`mise.toml`、`Cargo.toml` 等）是否暗示有未記載的慣例或工具
- **不要相信過時的記憶或 git log**，所有現況以當下讀檔結果為準

### 3. 編輯

- 維持原有章節順序、縮排、語系（繁體中文）與風格（`**粗體**` 標記關鍵詞、子項用 `-`）
- 局部更新用 `edit` 精準替換；不要整檔 `write` 覆蓋（會丟失使用者既有的自訂段落）
- 刪除過時段落直接移除，不要留「(已移除)」「(deprecated)」之類的痕跡
- 新增段落接在語意相近的章節下，不要塞到檔尾
- 目錄樹更新後，描述長度要與同層其他項目一致，避免一個項目寫 50 字、其他只寫 5 字

### 4. 驗證

- 重讀修改後的檔案
- 若有「專案結構」段落，實際 `ls` 對照一次，確認每個列出的目錄/檔案都存在且描述正確
- 確認章節順序、縮排、標點符號一致
- 確認沒有把應屬於 `README.md` 的內容塞進來

## 觸發情境的隱含提醒

- **主動詢問**：新增子目錄/工具/建構檔後，若判斷會影響 `AGENTS.md` 內容，主動問要不要同步
- **被動提醒**：發現 `AGENTS.md` 內容明顯與現況不符（例如列了不存在的目錄、漏掉新加的目錄），先指出來給使用者看，**不要擅自動手修**，等確認
- **不要為了「看起來完整」而新增空章節或套範本**；少而準確勝過多而模糊
- **不要在 `AGENTS.md` 內引用其他檔案的完整內容**；用一句話指明位置即可（例如「建構指令見 `package.json` 的 `scripts`」）
