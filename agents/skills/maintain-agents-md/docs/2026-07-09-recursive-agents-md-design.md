# maintain-agents-md: 加入遞迴 AGENTS.md 與註解分層 — 設計規格

**日期**: 2026-07-09
**作者**: brainstorming session
**狀態**: 待用戶審查
**目標檔案**: `agents/skills/maintain-agents-md/SKILL.md`

## 問題

現有 SKILL.md 主要聚焦於「專案根目錄的 AGENTS.md」,未明確處理:
1. 遞迴 AGENTS.md(子目錄內的 AGENTS.md)特性 — 子目錄 AGENTS.md 會被 AI agent 自動讀取,因此可以把更細的內容下移
2. 檔案內註解作為第三層抽象 — 針對單一檔案的細節應放註解而非 AGENTS.md
3. 三層(根 / 子目錄 / 註解)之間的內容應該如何分流

## 設計目標

- 保留「AGENTS.md 保持精簡」的大原則
- 把「精簡」具體化為「內容應放在最貼近讀者的層級」(三層分層)
- 讓 skill 同時指導 root 與子目錄 AGENTS.md 的建立與維護
- 加入「檔案註解」的選項,讓用戶知道細節可以再下沉

## 設計決策摘要

| 議題 | 決策 |

## 三層分層模型

| 層級 | 適用範圍 | 放什麼 |
|---|---|---|
| **根 `AGENTS.md`** | 全 repo 通用 | 跨多個子目錄、不屬於任何單一領域的規則;專案結構、整體慣例 |
| **子目錄 `AGENTS.md`** | 該子目錄(含其下所有檔案) | 該子目錄內跨檔案通用的設計原則、領域慣例 |
| **檔案內註解** | 該單一檔案 | 該檔案的特殊設計、不適用於其他檔案的細節 |

**判斷原則**:一段說明只適用一個檔案 → 放註解;適用於某子目錄所有/多數檔案 → 放該子目錄 AGENTS.md;適用範圍跨多個子目錄或全 repo → 放根 AGENTS.md。

````markdown
## 適用範圍

此 SKILL 處理的是**專案內所有層級**的 `AGENTS.md`(根目錄與子目錄皆含),給該 repo 內的 AI 編碼 agent 看的運作守則。不是 `~/.config/opencode/AGENTS.md`(用戶級全域規則)。

職責分離:
- 全域規則管所有專案通用行為
- 專案級 AGENTS.md 只描述此專案的特殊性
- 根與子目錄 AGENTS.md 共同遵守本 skill 的所有原則;差異在於適用範圍與具體章節內容

## 內容原則

- 給 agent 看的運作守則,不是給人類的 README;不寫教學、不寫廣告、不寫時效資訊
- 每行必須有資訊密度;agent 不靠 context 就能推論的內容不必寫
- 不與 `README.md` / `CONTRIBUTING.md` 重疊;README 講「這專案是什麼」,AGENTS.md 講「在這專案工作要注意什麼」
- 只寫變動頻率低的內容;太常變的東西會快速過時,會成為噪音
- 不要寫敏感資訊(token、內網位址、密鑰路徑個資等)

## 抽象層級規則(三層分層)

內容應放在「最貼近讀者」的位置。三層由外到內:

| 層級 | 適用範圍 | 放什麼 |
|---|---|---|
| **根 `AGENTS.md`** | 全 repo 通用 | 跨多個子目錄、不屬於任何單一領域的規則;專案結構、整體慣例 |
| **子目錄 `AGENTS.md`** | 該子目錄(含其下所有檔案) | 該子目錄內跨檔案通用的設計原則、領域慣例 |
| **檔案內註解** | 該單一檔案 | 該檔案的特殊設計、不適用於其他檔案的細節 |

**判斷原則**:一段說明只適用一個檔案 → 放註解;適用於某子目錄所有/多數檔案 → 放該子目錄 AGENTS.md;適用範圍跨多個子目錄或全 repo → 放根 AGENTS.md。

**反向檢查**(寫完後自我審查):
- 寫根 AGENTS.md 時發現某條只跟某個子目錄有關 → 下移到該子目錄 AGENTS.md
- 寫子目錄 AGENTS.md 時發現某條只跟某個檔案有關 → 改為該檔案的註解
- 反之亦然

## 建議章節(共同)

依照必要性取捨,不是每個 AGENTS.md 都需要全部:
- **章節概述**:該層 AGENTS.md 適用範圍一句話(子目錄尤其重要,讓讀者一眼看出「我在哪層」)
- **結構**:該層以下的目錄樹,每個項目附 5-15 字描述
- **慣例**:該層適用的命名、檔案組織、commit message 風格、模組邊界等
- **注意事項**:容易踩坑的地方、危險操作、非顯而易見的限制、`.gitignore` 排除但不查 git log 看不到的東西

不要的章節:安裝步驟、快速開始、貢獻指南、授權聲明——這些屬於 README。

## 根 AGENTS.md 專屬

根層通常多加:
- **專案概述**:一到三句話講清楚專案目的與邊界
- **工具特定子節**:例如 `commands/`、`scripts/` 的特殊使用方式(若僅在該子目錄內有效)

**根不該寫什麼 → 應寫到哪裡**:
| 不該出現在根 AGENTS.md | 應該寫到 |
|---|---|
| 只跟某個子目錄有關的設計原則 | 下移到該子目錄 AGENTS.md |
| 只跟某個檔案有關的細節 | 改為該檔案的註解 |
| 跨子目錄都適用的設計原則 | 留在根 AGENTS.md(這本來就是根的職責) |

## 子目錄 AGENTS.md 專屬

子目錄層**沒有固定必加章節**,視該子目錄特性決定。常用:
- **該子目錄的設計原則**(若跨多檔案通用;典型為 3+ 檔案)
- **該子目錄的領域慣例**(若非顯而易見)
- **該子目錄專屬的注意事項**

**子目錄不該寫什麼 → 應寫到哪裡**:
| 不該出現在子目錄 AGENTS.md | 應該寫到 |
|---|---|
| 跨子目錄或全 repo 通用的規則 | 上提到根 AGENTS.md |
| 只跟某個檔案有關的細節 | 改為該檔案的註解 |
| 該子目錄內跨檔案通用的設計原則 | 留在子目錄 AGENTS.md(這是它的職責) |

## 子目錄 AGENTS.md 何時該建

子目錄內出現以下任一情況,就值得建該層 AGENTS.md:
1. **跨 3+ 檔案通用的設計原則**(例:`src/components/` 的 components 通用設計原則)
2. **該子目錄有非顯而易見的領域慣例**
3. **該子目錄專屬的注意事項**(踩坑點、限制)

沒有上述情況,維持空白並把任何相關內容下放為檔案註解或上提到根 AGENTS.md。**不要為了「看起來完整」而建子目錄 AGENTS.md**;少而準確勝過多而模糊。

## 工作流程

### 1. 決定範圍
- **建立**:該層無 AGENTS.md 或用戶明確要求從頭寫
- **局部更新**:新增/移除單一目錄、慣例微調、注意事項新增/移除
- **全檔重寫**:使用者明確要求、或該層結構大幅變動
- **預設採用局部更新**,除非變動已影響超過約 40% 的內容

### 2. 盤點現況
- 用 `ls`、`tree`、glob 讀取該層以下的目錄結構
- 對照既有 AGENTS.md(若有)
- 檢查該層相關建構/設定檔(`package.json`、`pyproject.toml`、`mise.toml`、`Cargo.toml` 等)是否暗示有未記載的慣例或工具
- **不要相信過時的記憶或 git log**,所有現況以當下讀檔結果為準

### 3. 編輯
- 維持原有章節順序、縮排、語系(繁體中文)與風格(`**粗體**` 標記關鍵詞、子項用 `-`)
- 局部更新用 `edit` 精準替換;不要整檔 `write` 覆蓋(會丟失使用者既有的自訂段落)
- 刪除過時段落直接移除,不要留「(已移除)」「(deprecated)」之類的痕跡
- 新增段落接在語意相近的章節下,不要塞到檔尾
- 目錄樹更新後,描述長度要與同層其他項目一致,避免一個項目寫 50 字、其他只寫 5 字

### 4. 驗證
- 重讀修改後的檔案
- 若有「結構」段落,實際 `ls` 對照一次,確認每個列出的目錄/檔案都存在且描述正確
- 確認章節順序、縮排、標點符號一致
- 確認沒有把應屬於 `README.md` 的內容塞進來

### 子目錄流程差異

- **步驟 1 之前**:先判斷「該子目錄是否真的需要 AGENTS.md」(參考上方「子目錄 AGENTS.md 何時該建」三條件);若不需要,跳過建檔、改為下放檔案註解或上提根 AGENTS.md
- **步驟 4 額外檢查**:執行「反向檢查」(參考抽象層級規則),確認該層內容確實屬於此層,不應上提或下沉

## 觸發情境的隱含提醒

- **主動詢問**:新增/移除子目錄或工具後,若判斷會影響 AGENTS.md 內容(根或子目錄),主動問要不要同步
- **被動提醒**:發現 AGENTS.md 內容明顯與現況不符(例如列了不存在的目錄、漏掉新加的目錄),先指出來給使用者看,**不要擅自動手修**,等確認
- **不要為了「看起來完整」而新增空章節或套範本**;少而準確勝過多而模糊
- **不要在 AGENTS.md 內引用其他檔案的完整內容**;用一句話指明位置即可(例如「建構指令見 `package.json` 的 `scripts`」)
- **跨層調整時考慮三層分層**:新增內容時判斷該放哪層;既有內容不符時判斷該上提/下沉/註解
````
