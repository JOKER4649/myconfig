## 適用範圍

此 skill 處理 `opencode/agents/*.md` 下的 subagent prompt 檔案。每個 .md 是一個獨立 subagent,frontmatter 決定其 dispatch 介面(description、mode、model、permission),prompt 本體決定其行為。

**不處理**:
- `opencode/opencode.jsonc`(opencode 全域設定)
- `~/.config/opencode/AGENTS.md`(用戶級全域規則)
- 主 agent(如 build / plan)— 那些由 opencode 內建,不應該自建

## 核心方法論

### 1. description 是 dispatch 介面,不是 prompt 開頭

主 agent 看 description 決定怎麼呼叫這個 subagent。description 要明確說明:
- **定位**:這個 subagent 做什麼 / 不做什麼
- **適合呼叫**:什麼場景該呼叫
- **不適合呼叫**:什麼場景不該呼叫(改用其他 subagent 或主 agent 自做)
- **委派提示**:怎麼跟這個 subagent 互動(指定深度、模式、輸出格式等)

description **不要寫 prompt 本體才有的細節**(如視角定義、輸出 XML 範本) — 那些放在 prompt 本體。description 太長會稀釋 dispatch 關鍵字。

### 2. 任務輸入驗證 — 設計 subagent 的「拒絕機制」

subagent 通常需要拒絕模糊或超出職責的任務。設計時要明確:

- **必要要素清單**:這個任務要能執行,prompt 必須包含什麼?(每個 subagent 自行決定)
- **職責邊界**:subagent 不該做的場景是什麼?
- **失敗回應**:結構化錯誤格式(`Status: FAIL` / `Status: REJECT`)+ 補齊建議(讓 caller 知道怎麼改才能處理)

模糊 / 超出職責時 subagent 不該「猜測後執行」,猜測會引入偏差。

### 3. 輸出結構設計 — 思考消費端

subagent 輸出要讓呼叫者好消費。設計考量:

- **結構化 vs 自由**:結構化(易解析但較死板)vs 自由(易讀但難自動化)
- **平鋪 vs 屬性化**:每個結果帶結構化標籤(如時間、類型、來源),還是段落式整合?
- **跨維度資料怎麼表達**:一個事實有多個面向時 — 重複列出、用標籤(如 `[視角:X+Y]`)、或嵌套結構?
- **完整性驗證**:缺關鍵欄位算失敗?還是允許部分缺失?

核心原則:**設計跟消費端對齊** — 消費端是另一個推理 subagent 時,結構化 + 屬性化通常較好;消費端是人類時,自由格式較好讀。

### 4. 工具與資訊來源對應

subagent 的工具集決定它能獲得什麼資訊。設計時:

- **每個工具對應什麼資訊類型?**(本地檔案 / 網路 / 版本控制 / 計算)
- **任務特性 → 工具組合 → 輸出結構**:不同任務需要不同工具,prompt 內要明確指引
- **工具不足時的回應**:缺少必要工具時,subagent 該怎麼說?

### 5. 失敗 / 拒絕機制 — 給可執行的重派建議

任務出問題時,subagent 該如何回應?

- **結構化失敗格式**:一致的錯誤格式,讓呼叫者程式化處理
- **可執行的重派建議**:不只是「拒絕」,要告訴 caller 怎麼改派才能處理
- **灰色地帶處理**:任務部分超出職責時,處理可處理的部分 + 明確標示被忽略的部分,不要默默忽略

### 6. 跨 subagent 互動 — 設計時考慮上下游

若 subagent 輸出會被另一個 subagent 消費,輸出結構要對齊消費端的解析能力:

- 在 description 標註上下游關係(被誰呼叫 / 輸出給誰)
- 輸出結構要讓消費端能結構化解析(如 `<results>` / `<facts>` XML 標籤)
- 避免 subagent 間的語意斷鏈(如一個給「事實」,另一個給「結論」,中間沒人做推論)

### 7. 設計記錄 — 在 frontmatter 留 `note:` 區塊

每次設計要在 frontmatter 留 `note:` 區塊記錄:

- 為什麼這樣設計(決策理由)
- 替代方案是什麼、為什麼不選
- 已驗證 vs 未驗證假設
- 未來想改的人要知道什麼(已踩過的坑、跟其他 subagent 的關係)

這讓後續的 agent 或人能理解設計意圖,避免重複踩坑。

### 8. 避免邏輯規則殘留

重構 prompt 時容易留下舊版本的規則殘留(可能直接矛盾)。每次重構後:

- 列出所有「禁止」「必須」規則,確認它們邏輯一致
- 用 oracle 評估邏輯衝突
- 必要時跑實測看實際行為是否符合預期

## frontmatter 合規性

opencode agent frontmatter 合法欄位為:`name, model, variant, description, mode, hidden, color, steps, options, permission, disable, temperature, top_p`。

非官方欄位(如 `note:`)會被 silently 放進 `options` — 無功能影響(opencode 不會 hard-fail),但語意上變成「agent 配置」而非「文件元資料」。

**建議**:
- 給人看的設計記錄用 `note:` 欄位(可接受 silently 進 options)
- **不要把任何 subagent 行為依賴放進 `note:`**(LLM 不會讀到 frontmatter,行為約束要放 prompt 本體)
- 純時間戳不要獨立成欄位(資訊跟 git log 重複),時間資訊併入 `note:` 開頭一行(如 `最後更新:YYYY-MM-DD @ commit <hash>`)

## 建議章節結構

```yaml
---
description: |
  [定位 + 適合呼叫 + 不適合呼叫 + 委派提示]

mode: subagent | primary | all
model: ...
variant: ...
permission:
  [工具權限設定]
note: |
  最後更新:YYYY-MM-DD @ commit <hash>(<變更摘要>)
  [設計記錄:為什麼這樣設計 + 替代方案 + 假設狀態 + 未來注意]
---

## 職責
## 規則 (違反視為失敗)
## [核心概念:視角 / 探索深度 / 評估框架等]
## [工作方法]
## [輸出格式]
## 成功 / 失敗標準
## 停止條件
## 證據標準
## 職責護欄
```

章節順序可依 subagent 性質調整,不必每個都有。

## 工作流程

### 1. 釐清職責邊界

- 這個 subagent 做什麼 / 不做什麼?
- 對哪些現有 subagent 是替代或互補?(避免重疊)
- 在什麼模式下呼叫?什麼場景下呼叫?

### 2. 設計 description(給主 agent 看的 dispatch 介面)

精簡,只放 dispatch 決策需要的資訊。視角細節、輸出格式放在 prompt 本體。

### 3. 設計 prompt 結構

按「建議章節結構」順序填充。具體設計選擇(如視角、屬性化輸出)要根據 subagent 職責,不是通用模板。

### 4. 設計驗證標準

明確「成功 / 失敗」條件 — 成功是可驗證的正面行為,失敗是可列舉的負面行為。

### 5. 請 oracle 評估架構(可選但推薦)

大幅改寫前,先用 `task tool` 呼叫 oracle 評估架構設計。oracle 對 prompt 邏輯不嚴謻、未考慮到的邊界情況、內在衝突等特別敏銳。

提供給 oracle:
- 目前的設計跟替代方案
- 具體的設計問題(結構 / 命名 / 邊界)
- 期望的評估類型(架構性,不要「最佳實踐」)

### 6. 實測驗證

用 `opencode run --agent <name>` CLI 跑多場景實測。流程:
1. 暫時改 mode 為 `all`(subagent 不能直接用 `--agent` 跑)
2. 跑測試
3. 改回 mode 為原本的(`subagent` / `primary`)

每個測試要對應一個邊界或設計決策。關鍵測試類型:
- **正常任務**:覆蓋 subagent 主要職責的場景
- **模糊任務**:驗證拒絕機制,確認 subagent 不會自猜
- **超出職責**:驗證拒絕機制,確認給可執行的重派建議
- **不存在 / 錯誤假設的主題**:驗證不 hallucinate
- **跨維度資料(如適用)**:驗證跨維度表達機制
- **混合任務**:驗證灰色地帶處理
- **時效敏感主題(如適用)**:驗證時效欄位處理

### 7. 迴歸測試

prompt 大改後,跑之前驗證過的測試確認沒迴歸。重點:
- 之前能正確處理的核心場景
- 之前能正確觸發的 FAIL / REJECT 機制

### 8. A/B 評估(迭代必做)

迭代 prompt 時,**保留舊版**,用相同測試集跑新舊版,比較:
- 成功標準達成率
- FAIL / REJECT 觸發正確性
- 輸出結構合規率

用可列舉指標判斷這版是否比上版好,而非憑感覺。決定採用哪版後再 commit。

### 9. 維護

每次改寫寫好 commit message,記錄:
- 改了什麼
- 為什麼改
- 驗證了什麼
- 替代方案是什麼、為什麼選 A 不選 B

## 常見陷阱(反面案例,對應方法論/工作流程)

> 每條標「(對應方法論 X / 工作流程 Y)」,避免重複讀方法論論述。

### 描述層面

1. **description 過於詳細**(對應方法論 1):把 prompt 本體的細節塞進 description,稀釋 dispatch 關鍵字
2. **description 中兩個段落功能重疊**(對應方法論 1):如「應該呼叫 / 不要呼叫」跟「撰寫好的 prompt」,合併更精簡

### 任務輸入驗證

3. **缺乏任務輸入驗證**(對應方法論 2):subagent 收到模糊任務時不會拒絕,自猜執行
4. **缺乏失敗 / 拒絕機制**(對應方法論 5):subagent 默默失敗或無訊息拒絕
5. **邏輯規則殘留**(對應方法論 8):重構時舊規則沒清乾淨,跟新規則矛盾

### 輸出結構

6. **分類區塊對多面向事實硬傷**(對應方法論 3):同一事實屬於多個分類(如多視角、多類別、多來源)時,在多 section 重複或不一致 — 解法是「屬性化」或「標籤化」表達
7. **「其他」escape hatch**(對應方法論 3):分類值域加「其他」會被濫用,寧可誤標為次優值(有信號的誤標)
8. **屬性值域混維度**(對應方法論 3):把不同維度的值壓在同一個欄位會產生組合衝突 — 解法是每維度獨立欄位

### 模型行為

9. **依賴模型自發行為**(對應工作流程 6):某些行為是模型自發,換模型或換場景會消失。要靠 prompt 約束,不靠模型良心
10. **強制變體分支**(對應工作流程 6):對不同任務類型設多個變體分支會增加 prompt 複雜度,柔性引導通常較好

### 成本與擴展

11. **跨 instance / 跨 session 成本沒考慮**(對應工作流程 6):subagent 並行時成本累計,要在 description 或 prompt 提示
12. **沒留設計記錄**(對應方法論 7):未來想改的人不知道為什麼這樣設計

### 評估與驗證

13. **沒請 oracle 評估**(對應工作流程 5):大幅改寫前未讓 oracle 評估架構,容易留下邏輯衝突
14. **沒跑 A/B 評估**(對應工作流程 8):迭代時憑感覺判斷新版優劣,應保留舊版跑相同測試集比較
15. **沒跑實測驗證**(對應工作流程 6):純理論推理可能錯過實際行為偏差,實測是必要的

## 觸發情境

- 新增 subagent(對應新需求)
- 重塑現有 subagent 的職責 / 工具 / 輸出結構
- 調整 dispatch 行為(description 改寫)
- 修正 prompt 邏輯衝突(殘留規則相互矛盾)
- 修正 LLM 行為偏差(模型自發行為不可靠,需 prompt 約束)
- 整合新設計模式(如新的輸出結構、新的分類框架)
- 換模型 / 換 opencode 版本後重新驗證 prompt 相容性
